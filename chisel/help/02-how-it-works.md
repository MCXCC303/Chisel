# Chisel 是如何工作的

在开始之前，理解 Claude Code 的数据存储方式有助于更好地使用迁移功能。

Claude Code 的所有用户数据都位于 `~/.claude/` 目录下。每个项目的历史记录被组织在一个以编码路径命名的子目录中。例如，项目路径 `/home/alice/projects/my-app` 会被编码为 `-home-alice-projects-my-app`（简单来说，就是把路径分隔符 `/` 和特殊字符替换为 `-`）。

一个完整的项目记录包含以下部分：

### 会话文件
`~/.claude/projects/<encoded-name>/` 目录中，每个会话是一个以 UUID 命名的 `.jsonl` 文件。文件的每一行是一条消息记录，记录了用户输入、助手回复、工具调用等完整对话过程。

### 文件历史
`~/.claude/file-history/<uuid>/` 存储了会话中修改过的文件的快照。这对于追踪代码变更非常有价值，但并不影响对话本身的重现。

### 任务状态
`~/.claude/tasks/<uuid>/` 记录了会话中的任务列表进度。

### 全局索引
`~/.claude.json` 维护了每个项目的元数据：最后一次会话 ID、累计费用、权限配置等。`~/.claude/history.jsonl` 则是一个全局的命令行历史文件，记录了你在各个项目中输入过的所有指令。

### 迁移时的处理逻辑

**打包**时，Chisel 会将上述信息收集起来并做两件事：
1. 把项目绝对路径替换为占位符（如 `__CM_PROJECT_0__`），这样你的原始目录结构不会泄露
2. 将所有文件打包为一个 `.tar.gz` 归档

**解包**时，Chisel 会反向操作：
1. 让你为每个占位符指定新机器上的目标路径
2. 默认把 `/home/原用户名/` 替换为 `/home/当前用户/`
3. 将数据合并到已有的 `~/.claude/` 中

**智能合并**：如果目标机器上已存在相同 UUID 的会话，Chisel 不会简单地跳过或覆盖，而是逐行比对两个版本：

| 场景 | 处理方式 |
|------|---------|
| 包中的会话是目标版本的延续（目标的所有行都能在包中找到，且包有更多行） | **更新**为包中的版本 |
| 目标版本是包中版本的延续（包的所有行都能在目标中找到，且目标有更多行） | 保留目标版本（用户在目标机器上继续了对话） |
| 两边内容分叉（存在不一致的行） | **跳过**，保护已有数据 |
| 长度相同 | **跳过** |

这个策略确保新机器的对话进展不会丢失，同时旧机器的完整记录也能正确补充进来。
