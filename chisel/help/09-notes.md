# 注意事项

## 路径编码与碰撞

Claude Code 存在两套对特殊字符的编码：

| 位置 | 编码规则 | 示例 |
|------|---------|------|
| `claude.json` | 非 ASCII → `_` | `-home-alice-Applications-NextCloud-__-___` |
| 磁盘目录名 | 非 ASCII → `_`，**随后 `_` 又被替换为 `-`** | `-home-alice-Applications-NextCloud-------` |

这意味着同一个项目在 `.claude.json` 中记录的编码名和磁盘上实际的目录名**不同**。

目前认为这是 Claude Code 自身的设计缺陷：路径编码在不同代码路径中使用了不同的规则。

### 示例

假设你有下列项目：

```
/home/alice/Applications/NextCloud/课程/药理学			（3 个汉字）
/home/alice/Applications/NextCloud/课程/药剂学			（3 个汉字）
/home/alice/Applications/NextCloud/课程/形势与政策		（5 个汉字）
/home/alice/Applications/NextCloud/课程/生物药剂学		（5 个汉字)
```

磁盘上实际只会创建 **2 个目录**（而非 4 个），因为字符数相同的项目会碰撞：

|磁盘目录名                                          | 实际包含的项目|
|---------------------------------------------------|------------------|
|...NextCloud------- (7 个横线)                      | 药理学 + 药剂学（各 3 字）|
|...NextCloud--------- (9 个横线)                    | 形势与政策 + 生物药剂学（各 5 字）|


这导致在 Claude Code 使用时，会出现不同项目的会话混在同一个项目中的情况。

与此同时，`claude.json` 中记录的编码名又是另一种格式：
```
/home/alice/.../课程/药理学 → ...NextCloud-__-___  (下划线保留)
```

这使得磁盘目录和 JSON 记录**无法通过简单的字符串匹配对应起来**。

### 解决方法

Chisel 的扫描器会首先使用 `claude.json` 的编码规则尝试匹配磁盘目录，若失败，用磁盘规则再试一次。匹配成功后，通过 `history.jsonl` 中的 `project` 字段和会话 `cwd` 字段，将同一目录下混在一起的会话精确分发到正确的原始路径

综上所述，我们建议：

- **尽量使用纯 ASCII 符号路径**存放项目，避免中文、emoji 等非 ASCII 字符
- 如果无法避免，解包时把项目映射到不含中文的新路径

## 数据安全

### 清除操作不可逆
清除功能会从 `~/.claude/` 中彻底删除会话。**执行前务必**：
1. 使用预览功能确认每个会话的内容
2. 如果不确定，先打包备份
3. 仔细核对二次确认弹窗中的删除数量

### 打包前预览
打包前建议预览每个选中的会话，确保：
- 没有遗漏重要对话
- 没有包含不需要迁移的会话
- 确认会话内容与项目匹配（尤其是含碰撞路径的项目）

## 常见路径问题

### 修改项目位置后历史记录丢失
Claude Code 根据项目绝对路径关联历史记录。如果你**移动或重命名**了项目文件夹，原有的历史记录仍然存在于 `~/.claude/` 中，但 Claude Code 不会再主动显示它们。

**解决方案**：使用 Chisel 的打包功能导出历史记录，然后解包时指定新的项目路径。

### 跨机器迁移
从一台机器迁移到另一台时，Chisel 会自动将 `/home/原用户名/` 替换为 `/home/当前用户/`。如果两台机器用户名相同，路径完全不变。如果不同，在解包的映射步骤中可以手动修改。

## 性能

- 大型会话预览时可能较慢，Chisel 采用分批加载优化这一问题
- 扫描阶段读取 `claude.json` 和 `history.jsonl`，文件较大时（>1MB）初始化可能稍慢
- 打包大项目时建议预留足够的磁盘空间（通常为项目历史记录数据大小的 1-2 倍）
